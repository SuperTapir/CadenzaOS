# Cadenza：一台有 Playdate 气质的个人设备

## 项目名称

这台设备与它所代表的个人创作平台称为 **Cadenza**，系统称为
**Cadenza OS**，创作者署名为 **by Tapir**。

“Cadenza”原指乐曲中留给演奏者自由发挥的华彩段。这个名字对应项目的核心：
在统一而克制的软硬件结构中，为小游戏、工具、动画和交互实验留下个人表达的空间。
它也呼应旋钮或曲柄带来的节奏感与演奏感。

## 为什么要做它

这个项目最初来自对 Playdate 的喜爱。真正吸引人的并不只是“掌机”或“游戏平台”，而是它作为一件数字物件所呈现出的整体气质：

- 阳光下像纸一样清晰的 Sharp 黑白反射屏；
- 400 × 240、1-bit 画布带来的视觉克制；
- 高反差像素、抖动灰阶和复古图形；
- 富有弹性、节奏和触感的 UI 动画；
- 旋钮或曲柄带来的直接、连续的物理交互；
- 功能不多，但每个细节都经过设计。

因此，项目的目标不是复制 Playdate，而是创造一个**类似 Playdate 设计哲学的个人创作平台**：它拥有统一的 Launcher、应用结构、输入模型和视觉语言，可以承载小游戏、工具、动画与交互实验。这里的“平台”强调一套稳定、可持续扩展的软硬件规则，而不是追求海量游戏或兼容 Playdate 现有软件。

## 最终目标

最终设备将围绕现有的 Sharp `LS027B7DH01` 屏幕设计。这是一块 2.7 英寸、400 × 240 的黑白 Memory LCD，也是整个项目的视觉核心。

平台由四个彼此配合的部分组成：

- **设备**：围绕 Sharp 屏幕、旋钮或曲柄、按键与声音构成的实体；
- **系统**：负责启动、Launcher、设置、存储、输入和应用切换；
- **应用规范**：让每个小应用以统一方式启动、更新、绘制和退出；
- **设计语言**：400 × 240、1-bit 图形、像素字体、抖动灰阶和有触感的动画。

理想中的设备包括：

- Sharp 400 × 240 黑白反射屏；
- 一个具有鲜明视觉语言的 Launcher；
- 旋钮或曲柄式连续输入；
- 少量按键；
- 声音、存档和必要的无线功能；
- 若干个人化小应用、小游戏和互动玩具；
- 小巧、可握持、有完整外壳的产品形态。

它可能包含时钟、计时器、天气动画、每日文字、音乐界面和互动玩具，但不会为了数量而堆积功能。

## Cadenza OS 第一版：T-Embed 原型

第一版 Cadenza OS 使用 LILYGO T-Embed 作为软件和交互原型机。它已经集成 ESP32-S3、彩色屏幕、旋转编码器、按压输入、声音、MicroSD、电池电路和外壳，因此可以先避开焊接、飞线、面包板和复杂的电路排查。

T-Embed 不是最终硬件。它的作用是快速验证：

- Launcher 的结构和操作方式；
- 旋钮交互是否自然；
- UI 动画的节奏和手感；
- 应用的启动、退出和切换；
- 图形、字体、资源、设置与存档体系；
- 哪些功能值得进入最终设备。

开发时尽量使用黑白视觉，避免软件过度依赖 T-Embed 的彩色屏幕。

## 软件设计原则

### 1. 内容与硬件分离

应用不直接依赖具体屏幕或输入设备，而是通过统一接口使用画布和输入事件。

```text
Launcher / Apps / Visual Experiments
                 │
        UI、动画与应用框架
                 │
      逻辑画布与统一输入事件
          ┌──────┴──────┐
          │             │
     T-Embed 适配器   Sharp 适配器
```

### 2. Sharp 屏幕优先

最终目标画布为 400 × 240、1-bit。T-Embed 的实际屏幕为 320 × 170，因此原型阶段要明确处理缩放或预览方式，不让最终构图被临时屏幕绑死。

### 3. 交互优先于功能数量

优先打磨一个令人愉快的旋钮动画或页面切换，而不是快速加入大量普通菜单和工具。

### 4. 先内置应用，后动态扩展

第一版应用随固件一起编译，以稳定和简单为主。只有在 Launcher、输入和图形框架成熟后，才考虑从 MicroSD 加载资源或 Lua 应用。

### 5. 保持可迁移性

T-Embed 负责证明软件设计；Sharp 屏原型负责证明硬件；最终定制主板负责产品化。每个阶段都应尽可能复用前一个阶段的成果。

## 硬件路线

### 阶段一：T-Embed 软件原型

- 不焊接、不接线；
- 使用 USB-C 编译和烧录；
- 完成 Launcher 和第一批交互实验；
- 建立可迁移的软件结构。

### 阶段二：Sharp 屏幕验证

- 使用带 FPC 插座和正确供电电路的转接方案；
- 首先只验证测试图、帧缓冲和稳定刷新；
- 再移植 Launcher 和动画；
- 尽量采用工厂焊接好的转接板，避免手工飞线。

### 阶段三：完整定制载板

- ESP32-S3 模组或成熟核心板；
- Sharp 屏 FPC 插座；
- USB-C、电池和充电管理；
- 板载旋钮、按键和音频；
- PCB 厂完成贴片焊接；
- 用户只需插入屏幕排线、电池并安装外壳。

这里的原则是：不强迫项目变成电子焊接练习。可以通过模块化设计、工厂贴片和成品连接器，将精力留给软件、交互和产品设计。

## 第一阶段里程碑

1. 建立 Cursor + PlatformIO 开发环境；
2. 编译并烧录 T-Embed 的最小程序；
3. 验证屏幕、旋钮、按压和串口日志；
4. 完成黑白风格的三项目 Launcher；
5. 建立统一的应用生命周期；
6. 制作第一个旋钮驱动的互动场景；
7. 加入一个实用小应用；
8. 在电脑上提供 400 × 240 原尺寸预览；
9. 整理 Sharp 屏幕驱动与载板方案；
10. 将同一套 UI 迁移到真实 Sharp 屏幕。

## 当前明确不做的事情

- 不追求兼容或运行 Playdate 商业游戏；
- 不在第一版实现复杂的“应用商店”或操作系统；
- 不为了功能数量牺牲交互和动画质量；
- 不要求在开始软件开发前掌握完整的电子工程知识；
- 不把 T-Embed 当作必须沿用的最终硬件形态。

## 仍待探索的问题

- 最终使用旋转编码器、磁编码器还是机械曲柄；
- T-Embed 上采用等比例缩放还是局部预览 400 × 240 画布；
- Sharp 屏稳定刷新率和动画目标帧率；
- 最终设备需要哪些实体按键；
- 第一批应用的具体内容；
- 外壳尺寸、材料和屏幕保护方式；
- 完整载板是否采用 ESP32-S3 模组或可插拔核心板。

这份文档会随着原型推进持续更新。每次出现重要设计决定，都应记录“决定了什么、为什么这样决定、会影响哪些部分”。

## 已确定的软件约定（第一版）

- 应用实现统一的 `onEnter`、`onExit`、`update` 和 `render` 生命周期；
- 所有应用只绘制到 `MonoCanvas`，当前为 320 × 170 的 1-bit 内存画布；
- 输入在每帧被归一为旋转增量、短按、松开和长按事件；
- 短按由当前应用解释，非 Launcher 页面长按统一返回 Launcher；
- 第一阶段应用仍随固件编译，通过注册表加入系统，不从 SD 卡动态加载；
- 页面切换由系统层绘制统一转场，应用无需知道显示屏或转场细节。

这些约定优先验证“小型平台”的可行性。未来迁移到 400 × 240 Sharp 屏时，应用生命周期与输入模型保持不变，替换画布后端并逐步调整响应式布局。
