## 背景

`MonoCanvas::text` 能够根据当前 canvas clip 安全地测量和裁剪文本，但调用方只提供锚点、倍数和对齐方式。它无法推断真正的视觉容器，也不知道裁剪是否符合预期。因此，Launcher 会以 4 倍字号渲染所有当前应用名称，较长名称会穿过卡片内边距并最终被 framebuffer 裁剪；footer 的左右相邻应用名也没有独立宽度上限，存在侵占中央导航区域的风险。触发本次变更的截图中，`Animation Gallery` 已经被完全截断并产生 `text clipped`，这属于菜单布局失败，而不是内存安全问题。

解决方案横跨可移植布局、framebuffer 绘制、Launcher 呈现和测试。实现必须保留确定性的 1-bit 输出、固定容量行为、整数倍缩放，以及固件、headless 和桌面宿主之间一致的权威 framebuffer 数据。

## 目标 / 非目标

**目标：**

- 显式声明并测试文本的可用区域与溢出决策。
- 在不分配 heap 的前提下提供确定性的 `fit`、`ellipsis`、`wrap` 和 `marquee` 行为。
- 区分符合预期的呈现溢出与无效 geometry 诊断。
- 将动态 Launcher 标签迁移到受约束文本契约。
- 为菜单内的主标题、左右次要标签、边框和导航指示器建立明确且互不侵占的区域。

**非目标：**

- 增加比例字体族、Unicode shaping、双向文本或任意小数倍缩放。
- 在本次变更中自动迁移所有现有静态标签。
- 改变 320×170 或 400×240 framebuffer profile、已批准的应用截图语义或固件显示尺寸。
- 改变桌面调试 HUD 的覆盖式呈现、格式或开关行为。
- 引入通用的 retained-mode UI 框架。

## 技术决策

### 在 `MonoCanvas` 之上增加 presentation 层的受约束文本 API

新增 request/result 数据结构，而不是改变现有底层 `text` 调用的语义。request 包含文本、半开 bounds、对齐方式、首选/最小整数倍数、overflow/fallback policy、固定最大行数，以及显式的 marquee phase。result 报告状态、选定倍数、行片段、实际绘制边界，以及是否发生缩放、截断、换行或 marquee。

result 使用编译期有界存储保存行片段；片段引用输入文本区间，或引用 result 自己拥有的固定大小 scratch。可移植路径不允许使用 `std::string`、动态增长的 vector 或其他运行时分配。

备选方案：让 `MonoCanvas::text` 根据当前 clip 自动缩小。没有采用，因为 clip 是安全边界，不一定是真正的文本容器，也无法表达截断、换行或动画等产品决策。

### 分离布局与光栅化

布局阶段先测量并解析出具体呈现结果，随后由受约束绘制操作将结果光栅化到已解析矩形内。底层 `MonoCanvas::text` 保留现有的意外裁剪诊断。受约束绘制路径把有效 result 中声明的 viewport 裁剪视为预期行为，同时继续保护 framebuffer 写入，并诊断无效 request 或不一致 result。

这种分离使布局能够脱离像素检查进行测试，也避免仅靠 clip 掩盖问题、最终留下残缺文字。

备选方案：绘制裁剪文本时临时断开 diagnostic sink。没有采用，因为它会同时隐藏同一次操作中的真实存储或 geometry 错误。

### 使用确定性的策略顺序

`fit` 从首选倍数递减到最小倍数，选择第一个能够完整容纳文本的倍数。如果不存在完整容纳结果，则执行显式指定的 fallback：

- `ellipsis`：在最小倍数下选择能够容纳的最长前缀，并追加 `...`。
- `wrap`：尽可能在空白处分行；必要时硬拆过长 token；同时遵守垂直容量和固定最大行数；仍有剩余内容时，对最后一行使用省略号。
- `marquee`：保留请求倍数，解析 viewport 和内容范围，并仅根据调用方提供的归一化 phase 计算偏移。

辅助模块不读取时钟。动画拥有者可以使用模拟时间生成 phase，从而保持重放、seek、截图和测试的确定性。

备选方案：所有溢出都使用 marquee。没有采用，因为移动标签会降低可扫描性，也不适合密集的诊断信息。

### 根据菜单语义角色划分区域并选择策略

Launcher 主标题使用从 4 倍到 1 倍的 `fit`，随后以 `ellipsis` 兜底。其 bounds 是扣除双层边框、动画 kick 范围和文字 padding 后的卡片内部矩形，而不是完整 framebuffer。这样既能尽量完整显示 `Animation Gallery` 等已知名称，又能保证未来名称不会越界。

footer 被划分为左侧标签、中央导航指示器和右侧标签三个互不重叠的区域。左右次要标签在自己的区域内使用较小范围的 `fit`，随后以 `ellipsis` 兜底。布局不允许通过向 framebuffer 边缘对齐来代替区域约束，也不允许长标签侵占中央圆点。

后续菜单消费者复用同一模式：先根据视觉层级划分区域，再为每个动态文本 request 显式选择倍数范围和 fallback。静态、已知且有快照覆盖的标签无需在本次变更中强制迁移。

备选方案：只修复 `Animation Gallery`，将其字号从 4 倍硬编码为 2 倍。没有采用，因为它只能修复一个已知字符串，无法约束未来应用名、footer 标签或其他动态菜单数据。

## 风险 / 取舍

- [固定最大行数可能拒绝异常冗长的内容] → 在 result 中公开截断状态，并要求菜单为无法完整显示的内容选择明确 fallback。
- [减小标题倍数会改变已批准的 Launcher 快照] → 只更新此前代表意外裁剪的快照，并明确审查两个 profile。
- [为 footer 划分固定区域会减少左右标签的可用宽度] → 优先保证导航信息和元素隔离，再通过 `fit` 与 `ellipsis` 提供稳定降级。
- [换行规则可能膨胀成文本引擎项目] → 将本次变更限制在 ASCII 字体度量、空白/硬换行、整数倍数和固定行数内。
- [有意裁剪可能掩盖有价值的诊断] → 只抑制已编码在有效受约束布局 result 中的裁剪；其他路径继续保留底层诊断。

## 迁移计划

1. 为长标签、所有 overflow policy、无效 request 和 guard pixels 增加失败布局测试。
2. 实现固定容量 layout/result 模型和受约束光栅化，同时保留现有 `MonoCanvas::text` API。
3. 为 Launcher 主卡片和 footer 划分独立文本区域，迁移动态标签，并在像素审查后只更新受影响的快照。
4. 盘点其他菜单式动态文本调用，迁移确实接收运行时内容且缺少边界契约的调用点。
5. 运行可移植单元测试、快照测试和 desktop smoke 测试，并在两个 profile 下执行真实菜单视觉检查。

回滚范围是局部的：消费者可以恢复使用现有文本调用和区域常量，不会改变 framebuffer 存储或序列化数据。

## 待确认问题

- 首次实现需要通过视觉审查确定 footer 中央导航区域的最小保留宽度，并把该值记录为 Launcher 布局常量。
- `marquee` 保留为通用 overflow policy，但 Launcher 首次迁移是否实际使用它，由菜单静态可读性审查决定；默认优先 `fit` 与 `ellipsis`。
